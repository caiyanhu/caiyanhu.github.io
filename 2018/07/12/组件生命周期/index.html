Mounting

组件实例被创建并插入DOM时，下面函数按序执行：

  • constructor()
  • static getDerivedStateFromProps()
  • render()
  • componentDidMount()

避免使用

  • UNSAFE_componentWillMount()

Updating

props或state改变时会导致更新。当组件重新渲染时，按序执行下面函数：

  • static getDerivedStateFromProps()
  • shouldComponentUpdate()
  • render()
  • getSnapshotBeforeUpdate()
  • componentDidUpdate()

避免使用

  • UNSAFE_componentWillUpdate()
  • UNSAFE_componentWillReceiveProps()

Unmounting

当组件从DOM中被移除时，执行

  • componentWillUnmount()

Error Handling

当渲染过程中、生命周期函数中、子组件构造函数中出现错误时，执行

  • componentDidCatch()

函数详解

construtor()

constructor(props)

如果没有初始化state和绑定方法，那么不需要为组件实现constructor。

组件的constructor在组件完成挂载前执行。当实现子组件的constructor时，第一句必须是super(props);否则constructor中this.props就变成了undefined。

一般在constructor中做两件事情：

  • 通过this.state初始化local state
  • 绑定事件处理函数到组件实例上

不要在constructor()中调用setState();直接对this.state赋值就行。

constructor是唯一一处可以对this.state赋值的地方;在其它函数中应该使用this.setState()。

避免在constructor中引入有副作用的代码，应该把它们放在componentDidMount()中。

不要把props中的值拷贝到state中。

上面代码的问题是：既不需要这么做（可以直接使用this.props.color）;也容易出现问题（更新props下的color不会对state产生影响）。

只有在你想故意忽略props更新时再使用这种模式。

componentDidMount()

componentDidMount()

当组件挂载完成之后（插入到DOM树中），componentDidMount()函数立即被调用。在这个函数中可以：请求DOM节点、调用API接口来获取数据、订阅。

可以在componentDidMount()中立即调用setState()。该操作会触发一次额外的渲染，但此次渲染是在浏览器更新屏幕之前发生的，所以尽管多调用了一次render()，用户却对此一无所知。也因为会触发额外渲染，所以在使用时要注意性能问题。

componentDidUpdate()

componentDidUpdate(prevProps, prevState, snapShot)

当更新发生后componentDidUpdate()立即执行，但初次渲染时不会调用此函数。

可以在该函数中操作DOM，当props发生变化时可以进行网络请求。

当在componentDidUpdate()中立即调用setState()时要注意把它放在类似上面代码的条件下，否则将导致无限循环。此时调用setState同样会导致重新渲染，这个二次渲染同样对用户不可见，也同样小心性能问题。

如果你的组件实现了getSnapshotBeforeUpdate(),它的返回值将会作为componentDidUpdate()的第三个参数传入;否则，componentDidUpdate()第三个参数为undefined。

当shouldComponentUpdate()返回false时，componentDidUpdate()将不会被调用。

componentWillUnmount()

componentWillUnmount()

当组件卸载和销毁之前，componentWillUnmount()会被立即调用。在这里执行一些扫尾工作，例如注销定时器、取消网络请求、清理订阅。

不要在componentWillUnmount()里调用setState()，因为组件不会再次刷新了：一旦组件实例卸载了就不会被再挂载了。

shouldComponentUpdate()

shouldComponentUpdate(nextProps, nextState)

利用shouldComponentUpdate()来告诉React，当state或props变化时是否要更新组件。函数默认行为是：当state变化时重新渲染组件。

当props或state变化时、组件渲染前调用此函数;默认返回值为true;初次渲染时、使用forceUpdate()时不会调用此函数。

此函数可用于性能优化，但不能依赖它去阻止渲染。如果想跳过一些不必要的组件重渲染操作，可以考虑使用PureComponent。PureComponent只在自身state和接收父组件props变化时才会更新。PureComponent对props和state进行“浅比较”，减少了不必要的render操作次数。

如果要手写此函数，你应该比较props和state，返回false时告诉React不用更新组件。注意，返回false时只会影响当前组件，如果子组件的state变化了，子组件一样会重新渲染。

不建议在shouldComponentUpdate()中进行“深度相等性比较”或JSON.stringify()，因为这些操作比较低效，会影响性能。

如果shouldComponentUpdate()返回false，UNSAFE_componentWillUpdate()、render()、componentDidUpdate()都不会被调用。

static getDerivedStateFromProps()

static getDerivedStateFromProps(props, state)

就在render函数执行之前，getDerivedStateFromProps被调用，包括第一次挂载和后续的更新。此函数要么返回一个对象来更新state，要么返回null，什么也不更新。

这个函数适用于：当props会变化，而state又依赖于props时。

通过props来推导state会产生累赘的代码并让组件逻辑不再清晰。因此在使用此函数时注意：

  • 如果需要根据props的变化来执行一些副作用操作（数据获取、动画），使用componentDidUpdate
  • 如果只有当props变化时重新计算一些数据，使用memoization helper
  • 如果当props变化时重置一些state，可以考虑使用完全受控组件或带key的完全非受控组件

注意函数前面的static，这个方法不能被组件实例访问到。

每一次渲染都会导致此方法被调用，不管是因为何种原因。与之相对的是UNSAFE_componentWillReceiveProps，它只有在父组件导致的重新渲染（如props改变，而不是子组件的setState导致的更新）时被调用。

getSnapshotBeforeUpdate()

getSnapshotBeforeUpdate在新渲染结果生效之前被调用，此函数使得我们有机会在改变发生之前获取DOM的信息（如滚动条位置）。其返回值会作为componentDidUpdate()的第三个参数。

componentDidCatch()

componentDidCatch(error, info)

Error Boundaries是React组件，它能捕获子组件树中的错误，记录错误，展示后备UI而不是崩溃页面。Error Boundary能捕获发生在渲染期间、生命周期内、子组件的constructor中的错误。

当一个类形式的组件定义了该函数时，它就成为了一个Error Boundary。在该函数中调用setState()能捕获错误、展示备选UI。

在组件树中，Error Boundary只能捕获发生在子组件树中的错误，而不能捕获发生在自身的错误。

不建议使用的函数

UNSAFE_componentWillMount()

UNSAFE_componentWillMount()

UNSAFE_componentWillMount()在组合挂载的前一刻被调用。因为发生在render()之前，所以在这个函数里同步调用setState不会触发额外的渲染。

避免在这个函数中引入有副作用的代码，应该使用componentDidMount()。

UNSAFE_componentWillReceiveProps()

UNSAFE_componentWillReceiveProps

使用这个函数经常导致bug和不一致性问题，因此将来会被废弃。

如果在props变化时要执行有副作用的代码(数据获取、动画等)，使用componentDidUpdate

其他使用场景，参照这里

如果是因为“仅当props变化时要重新计算一些数据”而使用componentWillReceiveProps，建议使用 memoization helper

如果是因为“当props变化时重置一些state”而使用componentWillReceiveProps，建议换成 完全受控组件 或 带key的完全非受控组件

在非常罕见的情况下，可能会用到 getDerivedStateFromProps

当已挂载组件接收新的props之前UNSAFE_componentWillReceiveProps被调用。如果需要在props变化时更新state，需要在函数中比较this.props和nextProps，然后使用this.setState()进行状态变换。

注意，如果是因为父组件的原因导致子组件重新渲染，即使子组件本身的props没有变化，这个方法依然会被调用。

组件挂载期间，React不会调用UNSAFE_componentWillReceiveProps。调用this.setState()通常不会触发UNSAFE_componentWillReceiveProps。

此函数之前叫做componentWillReceiveProps，这个名称在React17之前会继续使用。

UNSAFE_componentWillUpdate()

UNSAFE_componentWillUpdate(nextProps, nextState)

在组件接收新的props或state之后，即将渲染之前，UNSAFE_componentWillUpdate()会被调用。初次渲染不会调用此方法。

不能在这个函数内调用this.setState()（否则会导致此函数被重复调用）;也不要在UNSAFE_componentWillUpdate`返回之前做会触发组件更新的操作（如dispatch redux action）。

这个方法可以被componentDidUpdate替换掉。

此函数之前叫做componentWillUpdate，这个名称在React17之前会继续使用。

如果shouldComponentUpdate()返回false，这个函数不会被调用。

其它API

上面的生命周期函数是React调用的，而下面的函数则是自己调用的。

setState()

setState(updater[, callback])

setState会将组件中state的变化加入队列，告诉React这个组件及子组件将会使用更新的state来重新渲染。这也是更新UI的主要手段。

将setState当作一次请求而非立即更新组件的命令。为了更好的性能，React可能会延迟执行，然后一次更新多个组件;而不保证state的变化会立即体现在UI中。

setState()可能会将state的变化打包或延迟执行，因此在setState()之后马上读取this.state可能不会得到期望的结果。其实应该在componentDidUpdate或setState()的回调中读取this.state，这两种方式均能保证读取到的state是变化之后的。如果新的state是基于老的state，看下面关于updater的介绍。

除非shouldComponentUpdate()返回false，否则setState()总会导致一次重新渲染。因此应该确保新旧state不周的时候再去调用setState()。

Q: 假如多次调用setState()，React更新组件的时候是按序调用这些setState()吗？

A:会！

Q: React何时会打包处理setState()?

A: 在React16及之前版本，默认只有事件处理器里的更新会被打包。在未来的版本里，React默认会打包所有更新。

在一次事件处理中，不管在多少个组件里调用多少次setState()，最终只会产生一次重新渲染。这对大型项目的性能尤其重要，毕竟如果处理一次点击事件时父子组件都调用了setState()，我们不希望子组件被渲染两次。

state的更新是按照setState的调用顺序进行浅合并。假如第一次更新是{a: 10}，第二次更新是{b: 20}，第三次更新是{a: 30}，那最后的state会是{a: 30, b: 20}。

在React16和之前的版本中，如果在React事件处理器之外，默认对setState()是不打包处理的。假如上面的代码不是在click的事件处理中，而是在ajax的回调内，就不会对setState进行打包处理了，每一个setState()会立即得到执行：

forceUpdate()

component.forceUpdate(callback)

默认情况下，当组件的state或props变化的时候，组件会重新渲染。如果组件的render()依赖其它的数据，可以通过调用forceUpdate()来告诉React这个组件需要重新渲染。

forceUpdate()会跳过shouldComponentUpdate()，直接调用render()。同时会触发子组件正常的生命周期，包括shouldComponentUpdate()。

尽量避免使用forceUpdate()。

实例属性

props

this.props包含了这个组件的调用者传入的值。

this.props.children是一个特殊的prop，用来表示JSX表达式中的子组件

state

包含了当前组件自身的状态，可能会随时间变化。state是自己定义的纯js对象。

如果一些值没有被用来渲染，那么没必要把它放进state。这样的值可以放入组件实例的字段中。
